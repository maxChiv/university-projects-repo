Group Twenty Two Project : Backend Documentation

Created by Eugene Yap, last modified on Oct 26, 2025

1. Overview

The RCPT backend provides the core business logic, data management, and API endpoints for the Research Costing and Pricing Tool — a web-based reimplementation of the University of Melbourne’s Excel-based RCPT workbook.
Its purpose is to let researchers and administrators:

  • Create and edit research project budgets (Staff Costs, Non-Staff Costs, Price Summary, etc.)

  • Automatically calculate full cost recovery, on-costs, and indirect rates

  • Manage project approvals (Head of Department → Dean → RIC)

  • Export approved budgets to PDF/CSV

  • Provide admin configuration of lookup data (salary tables, EBAs, multipliers)

2. Architecture Summary

Tech Stack

  • Language / Framework: Java 17 + Spring Boot

  • Persistence: JPA with hybrid database design:

    • PostgreSQL for lookup tables and structured reference data

    • MongoDB for flexible project JSONs and versioned costing data

  • Auth: Firebase Authentication (frontend) linked to Spring Boot backend

  • Build & Dependency: Maven (maven-compiler-plugin 3.14.0)

  • Deployment: Docker Compose with separate containers for backend, lookup service, MongoDB, and Postgres.

4. Domain Model (Simplified)

Based on the finalised UML:

Core Entities

  • User

    • Attributes: userId, name, email, isAdmin, isApprover

    • Relationships: controls multiple Projects.

    • Linked to Firebase UID for authentication.

  • Project

    • Contains:

      • ProjectDetails – title, funder, department, faculty, scheme, duration, currency.

      • StaffCosts – role, category, employment type, FTE/hourly, salary rate, cost recovery multiplier.

      • NonStaffCosts – cost type, isInKind flag, GST, indirect rate.

      • PriceSummary – total cost, currency, indirect multiplier, price to charge.

      • BudgetForm – summary of project data for approvals.

  • ApprovalTracker

    • Tracks approval status transitions (draft → submitted → approved).

    • Records actor, timestamp, and comments.

  • LookupDataHandler

    • Provides backend configuration (salary rates, EBAs, super, tax, cost recovery multipliers, etc.).

    • Admins can modify through a protected endpoint.

5. Authentication & User Management

  • Firebase Authentication provides token-based login from the frontend.

  • Backend validates tokens using Firebase Admin SDK.

  • Each user is persisted in Postgres with firebaseUid and role attributes (isAdmin, isApprover).

  • Role-based access control (RBAC):

    • Candidate (Researcher): create/edit projects

    • Approver: review and comment

    • Admin: modify lookup data and manage users

7. Database Design

MongoDB (Dynamic Documents)

  • Stores live project data and versioned costing JSON.

  • Suitable for variable structures (different staff/non-staff configurations).

  • Example:

    {
  "_id": "proj_123",
  "userId": "uid_abc",
  "projectDetails": {...},
  "staffCosts": [...],
  "nonStaffCosts": [...],
  "priceSummary": {...},
  "approvalStatus": "submitted"
}

PostgreSQL (Reference Tables)

  • Stores static, structured data used across all projects.

    • Salary bands

    • Cost recovery multipliers

    • Faculty departments

    • Research type lookup values

9. Deployment Notes

  • Docker Compose services:

    • backend → Spring Boot app

    • postgres → relational lookup data

    • mongo → project data

    • lookup-service → salary & multiplier sync

  • .env contains sensitive keys (Firebase, DB creds) — excluded via .gitignore

  • For deployment:

    1. Add .env file (Firebase + DB credentials)

    2. Run docker-compose up --build

    3. Verify backend at http://localhost:8080/api/v1/health
