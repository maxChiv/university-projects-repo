1. Introduction

This document outlines how to use the web-based Research Costing and Pricing Tool (RCPT) and guides how to continue developing the codebase. The client is The University of Melbourne Research, Innovation and Commercialisation (RIC) team. RIC team members are the admins of this website. It is created for researchers to form their project budgets. It was initially developed by a team of five university students.

At the outset, this project appeared daunting. The Excel workbook we were provided with to recreate presented significant complexity. It took us a lot of time to read through the manual and practice using the tool before we fully understood the logic and the requirements of our website - what to keep, what to get rid of, what to change. Then, after we had fully formalised our requirements sheet and broken the project down into tasks, we started to get to work on the website. Our main goals in designing this web based version of the RCPT were accessibility and simplicity. Each team member contributed consistently, and for much of the semester, we felt confident about meeting our goals. However, given the 12-week timeframe, we eventually had to make some difficult decisions around scope, prioritising essential features over nice-to-have functionality.

By the end of the semester, we successfully deployed a fully functional version of the RCPT that included the majority of its core features. This document serves as a handover guide for future developers, summarising the current state of the project and outlining how to build upon our existing foundation to complete the remaining components.

2. System Architecture & Design

The Research Costing and Pricing Tool (RCPT) is designed with a modular, service-oriented architecture that separates the system into distinct, maintainable layers — frontend, backend, and data services — all deployed through Docker.

The system follows a three-tier web architecture, consisting of:

  • Presentation Layer (Frontend):

    A React-based single-page application responsible for user interaction, data entry, and dynamic rendering of project details, staff costs, non-staff costs, and approval workflows. It communicates exclusively with backend APIs over HTTPS using JSON payloads.

  • Application Layer (Backend):

    A Spring Boot service that encapsulates the business logic, handles authentication and authorisation, performs cost and price calculations, manages approvals, and exposes RESTful endpoints for the frontend. The backend enforces data integrity through well-defined domain models, DTO mappings, and validation constraints.

  • Data Layer (Databases):

    The backend interacts with two persistent stores: a MongoDB instance that manages project-specific JSON data, and a PostgreSQL database that holds reference lookup tables, such as salary rates, cost recovery multipliers, and on-cost factors.

The design emphasises separation of concerns, scalability, and maintainability:

  • Each layer operates independently and communicates via well-defined APIs.

  • Lookup and calculation logic is externalised to allow policy updates without redeployment.

  • Docker Compose manages multi-container orchestration, ensuring consistent runtime environments across development, testing, and production.

This architecture supports both horizontal scaling (independent container deployment) and modular development, allowing future integration with enterprise systems (e.g. UoM authentication services) without major structural changes

2.3 Data Flow Overview

  1. User Interaction (Frontend)

    • The user fills in project details, staff costs, and non-staff costs through React forms.

    • Data is serialised as JSON and submitted via REST (e.g. POST /api/v1/projects).

  2. Backend API Handling

    • Spring Boot controllers (ProjectController, ApprovalController, etc.) receive the request.

    • JSON is converted into internal domain objects via MapStruct mappers.

  3. Business Logic Processing

    • CalculationService computes totals: salary + on-costs + multipliers.

    • PriceSummary and BudgetForm data are dynamically generated.

  4. Database Persistence

    • Processed data is stored in MongoDB as nested JSON documents.

    • Lookup data (salary tables, recovery rates) are retrieved from PostgreSQL via LookupService.

  5. Approvals and Workflow

    • When a user submits a project, ApprovalService updates its status (Draft → Submitted → Approved).

    • Approvers receive access to summary views or PDF exports.

  6. Output / Reporting

    • Frontend fetches the updated project with calculated fields.

    • Users can export results as PDF or CSV via API endpoints (/export/pdf, /export/csv).

Backend Tests:

Overview

Our backend code includes automated integration tests using JUnit. Tests support CRUD (Create, Read, Update, DELETE) operations, REST API mocking, persistence of data and check that JPA and H2 databases work as expected.

Tests

Each controller test consists of the same structure:

  • testGetAll()

  • TestGetBy…()

  • TestPost()

  • TestDelete()

Each creates data via JPA and pushes said data to an H2 database. By using H2, we can ensure that the system correctly hits the database without hitting the real database and risking affecting real data. Depending on the test, different CRUD operations are then performed by using MockMVC to mock a REST API to simulate real HTTP requests and hit the relevant endpoint.

Running the tests

  • Tests can be run locally by running ./mvnw clean test from the /Backend directory.

  • Alternatively, tests are run automatically when making a pull request or when pushing to dev and main branches. This is achieved via the CI workflow

MongoDB Testing:

Bring up the backend in Docker, verify it boots, then seed mock data and exercise the main approval + costing flows via HTTP. Each step includes the exact curl to run and the expected JSON response so you can confirm behaviour end-to-end.

9. Known Gaps & Future Work

9.1 Backend–Frontend API Integration

Overview

  • Goal: replace frontend mocks with the real backend API so the RCPT UI submits and receives canonical DTOs, backend pricing/approval/export logic runs on live data, and the frontend RCPT engine (rcptEngine) uses authoritative server responses rather than local fixtures.

  • Success will enable end-to-end project create/read/update flows, staff/non‑staff cost management, pricing summaries and approval actions against the backend.

  • Some progress has been made in the API-Integration branch

Current Status

  • Backend controllers and DTOs are implemented (examples: Project, Staff and Non‑Staff controllers, approvals, auth). See:

    • Backend project CRUD & export: ProjectController.java

    • Staff / Non‑staff endpoints: StaffCostController.java and NonStaffCostController.java

    • Auth: FirebaseAuthController.java

  • Backend DTO types (examples) to align with:

    • Project details and responses: Backend/.../dto/project/ProjectDetailsDto.java, ProjectResponse.java

    • Staff / NonStaff DTOs: StaffCostRequest.java, NonStaffCostRequest.java

    • Price summary / money / year allocation: PriceSummaryResponse.java, MoneyDto.java, YearAllocationDto.java

  • Frontend currently uses mock JSON fixtures and a legacy service layer:

    • Legacy service and DTOs: projectService.ts (many methods TODO and return *.json fixtures).

    • Local engine & cache that merges mocks and drafts: rcptEngine — loads fixtures via projectService, persists drafts in rcptCache and computes totals locally.

    • Dynamic forms and draft persistence: DynamicForm.tsx and rcptEngine.saveFormData mirror logic for canonical forms.

  • Preliminary wiring:

    • UI pages call rcptEngine and projectService (e.g., ProjectOverviewTab uses rcptEngine.updateProjectOverview and rcptEngine.saveFormData).

    • Config for backend base URL exists at config.ts (currently a default base, changeable for dev/prod).

  • Summary: backend business logic exists and is tested server‑side; frontend consumes mock fixtures and has TODOs to replace these with real endpoints.

Proposed Approach

  • Align frontend models & service signatures to backend DTOs

    • Update projectService.ts types and method return/request shapes to mirror backend DTOs (ProjectDetailsDto, StaffCostRequest/Response, NonStaffCostRequest/Response, PriceSummaryResponse / ProjectResponse).

    • Ensure field names, nesting and types line up (dates → ISO strings or structured date objects, Money → { amount, currency }, per‑year allocations → YearAllocationDto list).

    • Files to update: projectService.ts, any UI types imported from it, and usages in rcptEngine and form mappers.

  • Use a single service layer to isolate API mapping and HTTP concerns

    • Keep all HTTP calls, payload transforms and error handling inside projectService, userService and new staff/nonstaff services as needed.

    • Replace fixture reads (getJson(.../*.json)) with real HTTP calls using the configured base URL from config.ts.

  • Create lightweight handlers/adapters to translate between form shape and backend DTOs

    • Add small transformation functions that:

      • Validate and normalise form values (dates → LocalDate/ISO, money → MoneyDto, yearly maps → YearAllocationDto[]).

      • Strip UI‑only fields before sending.

      • Accept backend responses and map to frontend shapes expected by rcptEngine.

    • Place these adapters in projectService.ts or a new src/services/adapters/* folder.

  • Keep RCPT engine focused on snapshot, merge and totals

    • After service adapters return canonical DTOs, pass them to rcptEngine so it updates cache/totals and emits events.

    • Avoid moving HTTP or heavy mapping logic into rcptEngine—it should consume processed data only.

  • Auth, headers and user context

    • Replace the temporary user id ("1") in userService.ts / rcptEngine.updateProjectOverview with real auth flow: POST /api/auth/firebase-login (see controller) and include X-User-Id or Authorization headers as required by backend endpoints (controllers accept X-User-Id).

  • Endpoints & mapping (placeholders)

    • Project create/read/update: POST /api/v1/projects, GET /api/v1/projects/{id}, PUT /api/v1/projects/{id}

    • Staff: GET/POST /api/v1/projects/{projectId}/staff and POST /api/v1/projects/{projectId}/staff/append, DELETE /api/v1/projects/{projectId}/staff/{index}

    • Non‑staff: GET/POST /api/v1/projects/{projectId}/nonstaff and POST /.../append, DELETE /.../{index}

    • Pricing / export: GET /api/v1/projects/{projectId}/export and pricing endpoints (see ProjectController)

    • Auth: POST /api/auth/firebase-login

    • Approval actions: POST /api/v1/projects/{projectId}/approvals/{submit|approve|reject|comment}

Expected Outcome

  • Frontend forms and services fully connected to backend:

    • UI submits normalized DTOs and receives authoritative ProjectResponse / PriceSummaryResponse payloads from backend.

    • rcptEngine uses server data to compute/accept totals and keeps drafts in sync with server state.

  • Backend logic applied to live data:

    • Pricing, overhead, approvals, exports and other business logic run server‑side and are visible in the frontend.

  • Cleaner separation of concerns:

    • All HTTP, mapping and auth logic live in services; rcptEngine handles snapshot, merge, and totals only.

9.2 Caching Implementation

Overview

  • Goal: make the frontend caching mechanism deployable and reliable so project snapshots and form drafts persist across sessions/environments (not just in local dev). This could be a simplified way to get a proper MVP of the software.

  • Current behaviour: the frontend cache (sessionStorage + in-memory) implemented in rcptCache.ts works under npm run dev but is not surviving or behaving correctly after build/deploy (loss of drafts, inconsistent snapshot promotion, or mismatch across origins).

Proposed Approach

  • Use a deterministic, unique cache key

    • Extend the current key format in rcptCache.getEntryKey to include user context and a deployment/version identifier where available:

      • e.g. ${sessionKeyPrefix}:user:${userId}:project:${projectId}[:deploy:${deployId}]

    • Ensure the frontend supplies the authenticated user id (from the auth flow) when persisting/loading.

    • Files to change: rcptCache.ts, userService.ts.

  • Persist session/cache on the backend for durability

    • Add backend endpoints to persist and retrieve cache snapshots tied to the authenticated user (or session token). This prevents data loss when sessionStorage is cleared or the app is served from different origins.

    • Example behaviour:

      • POST /api/v1/sessions/cache — save snapshot (body: { userId, projectId, key, snapshot })

      • GET /api/v1/sessions/cache/{userId}/{projectId} — retrieve snapshot

      • DELETE /api/v1/sessions/cache/{userId}/{projectId} — clear snapshot

  • Wire frontend services to backend cache

    • Update projectService.ts to:

      • Check server-side cache before falling back to local sessionStorage.

      • Push snapshots to server after significant local changes (saveFormData / updateProjectOverview).

    • Update rcptEngine.ts to consult backend cache during loadData and to use canonical server snapshot when appropriate.

    • Ensure config.ts is used for base URL configuration (api.endpoint) and handles env overrides.

Expected Outcome

  • Persistent, deployable caching:

    • Drafts and project snapshots survive deployments and browser restarts when server cache is available.

    • Consistent cache keys tie snapshots to user+project+deploy, preventing cross-user leakage.

  • Improved UX and performance:

    • Faster project load when server snapshot exists; reduced reliance on public/mock fixtures.

  • Correct session handling:

    • Authenticated sessions used to scope cache access and protect user data.

9.3 Staff Cost Form

Overview

The current Staff Cost Form simply accepts a numeric input and assumes it represents the staff member’s salary. For accurate calculations, the form needs to account for employment classifications and types, linking to the Postgres lookup tables. Salaries should be derived from a time-based input (e.g., hours worked) rather than a direct monetary input.

Starting point: There is existing work on the branch fix/staff-cost-form which can be used as a reference for further development.

Current Status

  • The form currently accepts a number as the salary.

  • Backend logic to fetch classification-based salaries from the lookup tables exists but is not yet connected to the form.

  • Some preliminary adjustments to the dynamic form schema are present in the branch mentioned above.

Ideal Approach

  • Update the dynamic form schema to support multiple employment types and classifications.

  • Ensure the form calculates salary based on time input and the relevant lookup table values.

  • Connect the frontend form logic to the Postgres lookup tables to fetch salary rates dynamically.

  • Keep all computation logic within frontend services, passing final data to the backend or relevant engine as needed.

  • Reference the fix/staff-cost-form branch as a foundation for these changes.

Expected Outcome

  • Staff Cost Form calculates salaries correctly for different employment types and classifications.

  • Dynamic schema updates automatically based on employment type.

  • Frontend services handle all form logic and integrate seamlessly with backend data.

  • Form is fully tested and deployable in both development and production environments.

10. Conclusion

Our perspective on the project changed drastically week to week as we worked through the development process. At the very beginning we thought we were in for a very difficult semester, the original excel workbook RCPT appearing extremely complicated to replicate. However, during our first sprint - design - we started to fully understand the project we were working on and break it down into small tasks. We were very optimistic about the amount of features we could provide. Then, during the second sprint - coding - we started to visualise the amount of time each task took and how much time we had left. We had to make some cuts on features, including ones we had already put work into implementing. By the third sprint - deployment - we were still coding the last few features and spent a lot more time than anticipated on integration. Deployment was difficult but successful in the end and we are happy with the website we have provided. We hope this handover provides sufficient technical clarity and context for future teams to continue development and deliver a production-ready RCPT platform for the RIC.
