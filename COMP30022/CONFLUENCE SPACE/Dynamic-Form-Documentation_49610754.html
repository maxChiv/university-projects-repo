Group Twenty Two Project : Dynamic Form Documentation

Created by Max Chivers, last modified on Oct 04, 2025

Overview

DynamicForm renders a complete form purely from a JSON-driven FormSchema. It:

  1. Converts schema.fields → a runtime Zod object via buildFormSchema.

  2. Injects that Zod schema into React Hook Form through zodResolver.

  3. Renders each field using FieldForm, which delegates to concrete UI components (FormTextInput, FormSelect, etc.).

  4. Handles local submit state (saving) and calls the provided onSubmit with validated data.

  5. Supports repeatable (array) field groups.

Data Flow (JSON → Validation → UI → Submit)

  1. JSON FormSchema (fields + validation metadata).

  2. buildFormSchema(schema.fields) creates a Zod schema:

    • Each fieldDefinition.validation becomes chained Zod validators.

    • Repeatable fields become Zod.array(Zod.object(...)).

  3. useForm({ resolver: zodResolver(zodSchema), defaultValues }) wires Zod into React Hook Form.

  4. FieldForm maps field.type to a specific component and passes control + name.

  5. User input mutates RHF state; errors surfaced via field components.

  6. form.handleSubmit runs Zod validation; if success → handleSubmit → onSubmit(values).

  7. Caller performs persistence (API, mutation, etc.).

Validation Handling

Supported validation keys (see ValidationRules):

  • required → .min(1) / .nonempty() / .refine(...)

  • minLength / maxLength → .min(n) / .max(n) for strings

  • min / max → numeric boundaries for number types

  • pattern → new RegExp(pattern)

  • email → z.string().email()

  • url → z.string().url()
    Extend by:

  • Adding new keys to ValidationRules

  • Enhancing buildFormSchema to interpret them

  • (Optional) Updating field components for custom messaging

Key

Description

required

Enforces that a value must be provided.

minLength

Minimum number of characters (strings only).

maxLength

Maximum number of characters (strings only).

min

Minimum value for numeric fields.

max

Maximum value for numeric fields.

pattern

Regex pattern the value must match.

email

Must be a valid email format.

url

Must be a valid URL format.

Repeatable Arrays

Fields of type "repeatable" support dynamic sub-fields:

  • fields: nested FieldDefinition[] defining each sub-field.

  • defaultEntry: template object used when appending a new row/item.

  • Zod schema becomes z.array(z.object({ ...child schemas })).

  • UI: FormRepeatableArray uses useFieldArray for add/remove functionality.

    { "name": "startDate", "type": "date", "label": "Start Date", "validation": { "required": true } },
    { "name": "status", "type": "select", "label": "Status", "options": [
        { "value": "draft", "label": "Draft" },
        { "value": "active", "label": "Active" },
        { "value": "archived", "label": "Archived" }
      ],
      "validation": { "required": true }

      "name": "team",

      "label": "Team Members",
      "defaultEntry": { "fullName": "", "role": "" },

Example Usage in a Page / Component

import { DynamicForm } from "@/components/forms/DynamicForm"
import type { FormSchema } from "@/types/FormSchema"

const projectSchema: FormSchema = { ... } // same as JSON above

export default function ProjectCreatePage() {
  async function handleSubmit(data: any) {
    // Persist data via API call
    console.log("Submitted:", data)
  }

  const initialData = {
    title: "",
    status: "draft",
    team: [{ fullName: "Jane Doe", role: "pm" }]
  }

  return (
    <div className="max-w-2xl mx-auto">
      <DynamicForm schema={projectSchema} onSubmit={handleSubmit} initialData={initialData} />
    </div>
  )
}

Why This Approach is Best for Front-End Validation

  1. Single Source of Truth: The JSON schema defines both UI structure and validation rules. No duplication between front-end and backend validation.

  2. Type Safety: Zod generates runtime and TypeScript types, ensuring correct data shapes.

  3. Dynamic & Extensible: Adding new fields, types, or validations requires only changes in the JSON and schema builder.

  4. Centralised Validation: All validation runs before submit, catching errors immediately and providing consistent UX.

  5. Decoupled Persistence: The form handles only state, validation, and rendering. Backend integration is abstracted in onSubmit, keeping the component reusable.

This method ensures robust client-side validation, reduces potential backend errors, and allows dynamic form generation from JSON while remaining fully type-safe.

Adding a New Field Type

  1. Extend union in FieldDefinition.type (e.g. "radio").

  2. Implement UI component (e.g. FormRadioGroup) following existing component props signature.

  3. Add case to switch in FieldForm.

  4. Update buildFormSchema if validation semantics differ (e.g. multi-select arrays).

  5. (Optional) Add styling / tests.

Adding New Validation Rule

  1. Add key to ValidationRules (e.g. minItems).

  2. Update buildFormSchema to translate rule → correct Zod chain.

  3. Provide default error messaging or allow message override.

  4. (Optional) Add test coverage for rule application.

Conditional Visibility (Pattern)

Option A (filter before render):

{schema.fields
  .filter(f => (f.name === "budget" ? watch("status") === "active" : true))
  .map(f => <FieldForm key={f.name} field={f} control={form.control} />)}

Option B (add condition property):

  • Extend FieldDefinition with showIf?: (values) => boolean.

  • At render map: if (f.showIf && !f.showIf(form.getValues())) skip.

Grouped Sections / Steps

  1. Extend FormSchema with groups: { id, title, fields: string[] }.

  2. Render groups, mapping field names to field definitions.

  3. Maintain currentStep in state; validate per-step with trigger(groupFieldNames).

  4. Optional: Precompute a lookup: Map<fieldName, FieldDefinition> for O(1) access.

Repeatable Field Tips

  • Provide stable key: use field.id from useFieldArray (inside FormRepeatableArray).

  • defaultEntry should include every nested field for predictable dirtiness / validation states.

  • To enforce at least one member: add validation rule in builder (e.g. team: z.array(...).min(1)).

Error Handling & UX

  • Field components should read formState.errors[name].

  • Consider a global error summary (Object.keys(errors)) above the form.

  • Disable submit while saving to prevent duplicate calls.

Performance Notes

  • zodSchema memoized with useMemo on schema.fields reference.

  • Avoid regenerating schema by mutating fields array in-place—create new arrays when updating.

  • Lazy-load heavy field components if needed.

Testing Strategy

  • Unit test buildFormSchema: given field defs → expected shape / validators.

  • Render DynamicForm with testing library:

    • Fill inputs → submit → assert onSubmit receives expected object.

    • Trigger validation errors.

Extensibility Checklist

  • Field Types: Update union + renderer.

  • Layout: Wrap groups / steps.

  • Accessibility: Add aria-describedby with error IDs, role="alert" for error messages.

  • Internationalization: Provide label/message keys instead of raw strings.

Minimal Quick Start

  1. Define FormSchema object.

  2. Pass to DynamicForm with onSubmit handler.

  3. (Optional) Provide initialData for edit mode.

  4. Handle persisted response (toast, redirect, etc.) outside the form.

Common Pitfalls

  • Forgetting required:true in validation → field remains optional.

  • Supplying options without value/label shape.

  • Mutating schema.fields directly (breaks memoization).

DynamicForm Styling Report

Current Styling Sources

  • Layout container: The <form> in DynamicForm uses Tailwind class space-y-6 (vertical rhythm only).

  • Horizontal padding: Supplied by the optional <Card> wrapper (p-6 default) in DynamicForm.

  • Field wrapper: Each rendered field is enclosed by a <div class="field-form-wrapper"> in FieldForm with data attributes (data-field, data-type, data-path) serving as styling / theming hooks.

  • Field-level appearance: Resides inside individual components (e.g. FormTextInput, FormSelect, FormNumberInput, etc.) imported by FieldForm. These ultimately use shared UI primitives (e.g. Input in Frontend/src/components/ui/input.tsx).

  • Action section: Submit button area is a plain FormItem + Button (no extra layout constructs).

Separation of Concerns

  • DynamicForm: Minimal structural spacing + optional card chrome.

  • FieldForm: Provides a neutral, predictable wrapper for every field type without imposing layout (no margins applied—relies on parent spacing).

  • UI primitives: Own typography, borders, focus states (centralized theming surface).

Extension / Future Directions

  1. Grid / Multi‑Column Layout

    • Replace space-y-6 with a responsive grid (e.g. grid md:grid-cols-2 gap-y-6 md:gap-x-8).

    • Add optional schema hints like layout?: { columns?: number } or per-field span?: number.

  2. Field Spanning

    • Support fullWidth or colSpan in FieldDefinition and apply conditional classes (e.g. md:col-span-2) on the wrapper.

  3. Theming Hooks

    • Leverage existing data attributes: [data-type="number"], [data-type="repeatable"] for contextual borders, badges, or background.

    • Add data-required when validation.required is true for visual indicators.

  4. Variant Prop / External Class Injection

    • Add formClassName and fieldsWrapperClassName props to DynamicForm to allow caller-controlled layout without modifying internals.

  5. Group / Section Styling

    • When groups/steps are introduced (future schema extension), wrap groups in a container with heading + space-y-* isolation; apply progress or stepper UI.

  6. Error & State Styling

    • Introduce a form-level error summary zone (aria-live) above fields.

    • Add conditional border or iconography on .field-form-wrapper when the field has an error (derive via RHF formState.errors).

  7. Theming System

    • Abstract frequently reused spacing/padding into design tokens or Tailwind config.

    • Introduce light/dark mode variants via data-theme on the outer container.

Quick Win Changes (Non-Breaking)

  • Add optional prop: layout="stack" | "grid" (default stack). If grid → apply grid classes.

  • Pass a fieldWrapperClassName to merge onto the .field-form-wrapper.

  • Add a boolean condense prop to reduce vertical rhythm by switching space-y-6 to space-y-4.

Risks / Considerations

  • Hardcoding grid now without schema hints may force awkward spans later.

  • Adding margins inside field components can fight the parent spacing model—keep spacing external.

  • Overusing per-field inline classes can fragment theming; prefer centralized wrappers + data attributes.

Example Usage: ProjectFormPage

File: ProjectFormPage

Purpose

Demonstrates end‑to‑end usage of the generic DynamicForm with:

  • A remotely loaded JSON form schema (/forms/projectForm.json)

  • Prefetched initial entity data from a demo service (ProjectService)

  • A persistence handler (handleSubmit)

Lifecycle & Data Flow

  1. Mount effect (useEffect):

    • Fetches the JSON schema: fetch("/forms/projectForm.json")

    • Fetches existing project data via ProjectService.getProjectData (used as initialData)

    • Stores both in local state: schema, initialData

    • Clears loading flag when done.

  2. Conditional render:

    • While loading or schema missing → displays a simple Loading... placeholder.

  3. Render of DynamicForm:

    • Props passed:

      • schema: the fetched JSON (drives field rendering + Zod schema generation internally)

      • initialData: pre-populates form values (edit mode)

      • onSubmit: async function calling ProjectService.saveProjectData

  4. On submit:

    • DynamicForm validates via its generated Zod schema

    • If valid, it invokes handleSubmit

    • handleSubmit saves data, then logs: Saved project form: ...

What This Shows

  • Decoupling: Page never touches individual inputs—only supplies schema + data.

  • Reuse: Any other page can replicate this by swapping the fetched schema + service.

  • Edit Mode: Passing initialData demonstrates how existing records are loaded for modification.

  • Async Gate: Simple loading state ensures the form isn’t initialized before schema exists.

Key Props in Action

  • schema: Drives which components FieldForm renders (text, number, select, repeatable, etc.).

  • initialData: Becomes defaultValues inside useForm (in DynamicForm).

  • onSubmit: Is wrapped by DynamicForm’s internal handleSubmit to manage its local saving state.

Minimal Mental Model

JSON schema → (page loads) → pass to DynamicForm → DynamicForm builds Zod → React Hook Form manages state & validation → user edits → submit → validated data forwarded to page’s handleSubmit.

Possible Enhancements (not implemented here)

  • Error boundary / toast on fetch failures

  • Loading skeleton instead of plain text

  • Optimistic UI or dirty-state detection

  • Schema/version caching

import { useEffect, useState } from "react"
import { DynamicForm } from "@/components/forms/DynamicForm"
import type { FormSchema } from "@/types/FormSchema"
import { ProjectService } from "@/services/ProjectServiceDemo"
import type { ProjectData } from "@/services/ProjectServiceDemo"

export function ProjectFormPage() {
  const [schema, setSchema] = useState<FormSchema | null>(null)
  const [initialData, setInitialData] = useState<ProjectData | null>(null)
  const [loading, setLoading] = useState(true)

  useEffect(() => {
    async function load() {
      try {
        const s = await fetch("/forms/projectForm.json").then(r => r.json())
        setSchema(s)
        const data = await ProjectService.getProjectData()
        setInitialData(data)
      } finally {
        setLoading(false)
      }
    }
    load()
  }, [])

  async function handleSubmit(data: any) {
    await ProjectService.saveProjectData(data as ProjectData)
    console.log("Saved project form:", data)
  }

  if (loading || !schema) return <p>Loading...</p>

  return (
    <div className="p-8 flex justify-center">
      <DynamicForm schema={schema} initialData={initialData || undefined} onSubmit={handleSubmit} />
    </div>
  )
}
