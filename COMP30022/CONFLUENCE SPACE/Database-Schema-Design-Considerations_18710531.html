Group Twenty Two Project : Database Schema Design Considerations

Created by Eugene Yap on Sept 04, 2025

The hybrid method (SQL for lookup tables, JSON for project data and flexible sections) is not only good, it matches both the RCPT’s original design intent and the domain model.

  1. Lookup tables are the backbone of RCPT
    The user manual and functional requirements highlight that salary rates, cost recovery multipliers, departments, schemes, currencies, etc. are all maintained as “lookup tables”. These need to be reliable, queryable, and updateable by admins, which makes a relational SQL structure ideal.

  2. Flexible project and costing data
    The domain model shows entities like ProjectDetails, StaffCosts, NonStaffCosts, PriceSummary, ApprovalTracker. Many of these contain funder-specific or department-specific fields (e.g., in-kind costs, indirect multipliers, deliverables). Requirements mention undo/back and “staged JSON versions” for projects. That’s a clear case where JSON storage is more practical than endlessly altering SQL schemas.

  3. Compliance & workflow
    Approvals, roles (admin/approver/user), and authorisation flows are well suited to SQL tables with clear FKs and audit logs. But attaching comments, notes, or metadata to these records is lighter with JSON.

  4. Future-proofing
    The RCPT manual stresses that funding rules change frequently (salary increments, indirect rates, scheme-specific rules). A hybrid model means your lookup SQL tables handle stable, structured rules, while JSON fields handle evolving or scheme-specific extras without schema rewrites.
