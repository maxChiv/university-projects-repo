Decision 1 — UI Component Library Selection

Context:

The research costing and pricing tool requires a consistent, accessible UI library with form components, data tables, and interactive elements. Need rapid development capability while maintaining design consistency across the application.

Options Considered:

  • Option A — Build custom components from scratch

  • Option B — Use shadcn/ui as primary library with Radix primitives

  • Option C — Use alternative libraries like Chakra UI or Material-UI

Decision:

  • Use shadcn/ui as the main component library

  • Implement custom wrappers around shadcn components for form handling

Consequences:

  • Rapid development of forms, dialogs, tables, inputs with consistent styling

  • Accessible components by default

  • Need to learn shadcn/ui patterns and Tailwind utility classes

  • Limited to Tailwind's core utility classes (no custom compilation) which will keep the design consistent.

Decision 2 — Form Architecture

Context:

The application requires complex forms for project details, staff costs, equipment, and other research costing inputs. Need efficient validation, error handling, and state management across multiple form sections.

Options Considered:

  • Option A — Manual form state with useState for each field

  • Option B — React Hook Form with shadcn Form components and Zod validation

Decision:

  • Use React Hook Form for performance (uncontrolled inputs, minimal re-renders)

  • Use shadcn's Form, FormField, FormItem, FormControl, FormMessage primitives

  • Use Zod for validation and type derivation

  • Create reusable field wrapper components (FormTextInput, FormSelect, FormTextarea)

Consequences:

  • Excellent performance with uncontrolled inputs

  • Consistent styling and accessibility via shadcn primitives

  • Strong type safety with Zod schemas

  • Reusable components reduce code duplication

  • Learning curve for React Hook Form patterns

Notes:

Documentation can be found here: https://ui.shadcn.com/docs/components/form

→ Reference this for any knowledge gaps. There are some good examples of Text Input Forms

Structure of the shadcn Form wrapper is:

  • Form = context provider for RHF state.

  • FormField = connects UI → RHF for one field.

  • field = props from RHF (value, onChange, onBlur) → binds Input to state and can be changed to implement different states.

  • onSubmit = gets all form data at once no need to manually track useState.

By creating our own wrappers for inputs with Shadcn Forms allows us to build quickly by using only set out and well tested forms that we have made.

It will give rise to a simple Data Flow. Here’s the end-to-end flow when a user types into your form:

  1. User types into <Input> → RHF updates field state.

  2. Submit button pressed → onSubmit(values) runs.

  3. Save the data, an option could be:

    1. (TBD) setProject(values) updates global context → any other page or component reading from it updates instantly.

    2. (TBD) saveProjectDetails(values) sends JSON → backend API.

  4. Backend saves → returns confirmation or new data.

  5. (Optional) Context updates with backend response if needed.

Going through an example and breaking it down:

This is a reusable input wrapper around ShadCN + RHF so you don’t repeat FormItem boilerplate everywhere. It is not to be used for usage, as we will see how to construct the actual component we can use. 

type TextInputProps = {
  name: string;             // field key in the form data object
  label: string;            // label shown to user
  placeholder?: string;     // placeholder text
  required?: boolean;       // optional HTML validation
  control: any;             // the RHF "brain" controlling this field
};

export const FormTextInput = ({
  name,
  label,
  placeholder,
  required,
  control,
}: TextInputProps) => (
  <FormField
    control={control}         // connect to RHF state manager
    name={name}               // register this field under that name
    render={({ field }) => ( // field = hooks to read/write state
      <FormItem>
        <FormLabel>{label}</FormLabel>
        <FormControl>
          <Input {...field} placeholder={placeholder} required={required} />
        </FormControl>
        <FormMessage />       // show validation errors for this field
      </FormItem>
    )}
  />
);

→ properties you pass into component:

  • control: comes from useForm() (RHF). So when you pass <FormField control={form.control} name="title" /> youre telling RHF that “This input should be registered under the name title, and I want you to manage its value, validation, and lifecycle.”

  • name: the key name in your form data ("projectName", "description", etc.).

  • label: user-facing label text.

  • placeholder: optional hint text.

→ the render callback gives you a special field object:

field = {
  onChange: (e) => void,   // Updates RHF state
  onBlur: () => void,      // Marks field as touched
  value: string,           // Current field value
  ref: Ref<HTMLInputElement>  // So RHF can focus, validate, etc.
}

We can then use this to make a form with values that can be used to actually obtain data and do stuff with it automatically:

// components/forms/ProjectForm.tsx
import { useForm } from "react-hook-form";
import { TextInput } from "@/components/forms/TextInput";
import { Button } from "@/components/ui/button";
import { useProject } from "@/context/ProjectContext";

type ProjectFormValues = {
  title: string;
  description: string;
};

export const ProjectForm = () => {
  const { updateProject, saveProject } = useProject();
  
  const form = useForm<ProjectFormValues>({
    defaultValues: {
      title: "",
      description: "",
    },
  });

  // 1. Generalised onSubmit: update context instead of saving directly
  const onSubmit = (values: ProjectFormValues) => {
    updateProject(values);     // merge values into global state
    console.log("Saved locally:", values);
  };

  return (
    <Form {...form}>
      <form onSubmit={form.handleSubmit(onSubmit)} className="space-y-4">
        <TextInput control={form.control} name="title" label="Project Title" placeholder="My Research Tool" />
        <TextInput control={form.control} name="description" label="Description" placeholder="Brief project summary" />

        <div className="flex gap-4">
          <Button type="submit">Save Locally</Button>
          <Button type="button" onClick={saveProject} variant="secondary">
            Save to API
          </Button>
        </div>
      </form>
    </Form>
  );
};

Decision 3 — Use Project Context as an Intermediate Layer

Context:

We want form data to be accessible globally in the app (across pages and components) without prop drilling. The backend API may not be ready yet, or we may want to decouple UI logic from persistence.

Options Considered:

  • Direct API calls from the form: each form field would save directly to the backend.

  • Local state only: store values in the component; other pages/components cannot access it.

  • Global Project Context: store all form data in context; UI reads/writes from it, API can be added later.

Decision:

Use a Project Context (ProjectContext) to store all project-related form data.

Consequences / Benefits:

  • UI decoupled from persistence: forms and components don’t care if data is in memory or sent to API.

  • Global access: any page/component can read or modify project data.

  • !! Easy API integration later: only ProjectContext needs modification; forms remain unchanged.

  • Immediate reactivity: components reading context re-render automatically when data changes.

Notes:

The flow will look something like this, where anything to do wiht context or data flow is from the ProjectContext file:

User Input → Form Fields → onSubmit → updateProject() → ProjectContext
                                   ↓
                         Optional API saveProject()

We can make sure this is the one truth for data integration, keeping it seperate from everything for now.

!! IMPORTANT NOTE: it must match the same structure as the api is expecting (as in the JSON structure).

A brief example of how this project context structure will integrate with a Form:

A content file ProjectContext.tsx:

type ProjectData = {
  name: string; // example field; can expand with more fields like author, tasks, etc.
};

type ProjectContextType = {
  project: ProjectData;
  setProject: (data: ProjectData) => void;
};

const ProjectContext = createContext<ProjectContextType | undefined>(undefined);

export const useProject = () => {
  const context = useContext(ProjectContext);
  if (!context) throw new Error("useProject must be used inside ProjectProvider");
  return context;
};

export const ProjectProvider = ({ children }: { children: ReactNode }) => {
  // Global state for project
  const [project, setProject] = useState<ProjectData>({ name: "" });

  return (
    <ProjectContext.Provider value={{ project, setProject }}>
      {children}
    </ProjectContext.Provider>
  );
};

Explanation:

  • ProjectContext → holds the global project state.

  • useProject() → custom hook to access context anywhere.

  • setProject() → function to update the project globally.

A simple form using the Project Context to save its data:

export const SimpleInputForm = () => {
  const { setProject } = useProject(); // access global state

  // create form with React Hook Form
  const form = useForm<{ name: string }>({ defaultValues: { name: "" } });

  // onSubmit updates global project context
  const onSubmit = (values: { name: string }) => {
    setProject(values);       // << merge typed value into global state
    console.log("Saved:", values);
  };

  return (
    <Form {...form}>
      <form onSubmit={form.handleSubmit(onSubmit)}>

        {/* Single input field */}
        <FormField
          control={form.control}   // connects input to form state
          name="name"
          render={({ field }) => (
            <FormItem>
              <FormLabel>Your Name</FormLabel>
              <FormControl>
                <Input {...field} />  {/* controlled input */}
              </FormControl>
              <FormMessage />        {/* shows validation errors */}
            </FormItem>
          )}
        />

        <Button type="submit">Save</Button>
      </form>
    </Form>
  );
};

Explanation of the Flow:

  1. User types → Form updates RHF internal state.

  2. User submits → handleSubmit gathers the current field values.

  3. onSubmit receives { name: "typed value" }.

  4. Call setProject(values) → updates the global context state.

  5. Any component using useProject() can now read the updated project.name.

Decision 4 — Dynamic Form Rendering Strategy

Context:

Multiple pages require similar form structures with different field configurations. Need to avoid duplicating form rendering logic while maintaining flexibility to modify form schemas without code changes. Forms should support basic fields, repeatable sections, and validation.

Options Considered:

  • Option A — Hard-coded forms for each page with duplicated rendering logic like a tsx component per form.

  • Option B — Full block-driven system with complex metadata and layout management

  • Option C — Schema-driven forms with page-level JSON configurations and shared DynamicForm renderer

Decision:

  • Implement schema-driven forms with per-page JSON configurations

  • Create a lightweight DynamicForm component that renders fields based on schema

  • Keep schemas data-focused (fields + validation) without layout information

  • Use existing shadcn field wrappers (FormTextInput, FormSelect, FormNumberInput)

  • Support repeatable sections using React Hook Form's useFieldArray

  • Page components control layout and positioning of forms

Consequences:

  • Low ceremony approach - easy to add or modify forms via JSON configuration

  • Consistent UI through reusable field wrappers and DynamicForm renderer

  • !! Non-technical users can modify form questions by editing JSON schemas

  • Reduced code duplication across similar form pages

  • Maintains flexibility to upgrade to full block system later

  • Requires maintaining schema versions for data migration

  • Initial overhead to build DynamicForm renderer and schema structure

Notes:

Schema files will be stored in schemas/ directory (e.g., projectInfo.schema.json, staffCosts.schema.json). DynamicForm handles field rendering and validation, while page components control layout decisions. Supports both simple fields and repeatable arrays for dynamic content like staff lists or equipment items.

A high-level, is this

An example of two json files which could be used to load forms with id, names, and type of Form:

schemas/projectInfo.schema.json

{
  "id": "project-info.v1",
  "title": "Project Info",
  "fields": [
    { "id": "title", "type": "text", "label": "Project Title", "required": true },
    { "id": "author", "type": "text", "label": "Author" },
    { "id": "summary", "type": "textarea", "label": "Summary" }
  ],
  "repeatables": []
}

schemas/staffCosts.schema.json

{
  "id": "staff-costs.v1",
  "title": "Staff Costs",
  "fields": [],
  "repeatables": [
    {
      "id": "staffCosts",
      "title": "Team members",
      "itemSchema": [
        { "id": "name", "type": "text", "label": "Name", "required": true },
        { "id": "role", "type": "text", "label": "Role" },
        { "id": "hours", "type": "number", "label": "Hours", "min": 0 },
        { "id": "rate", "type": "number", "label": "Rate", "min": 0 }
      ]
    }
  ]
}

This demonstrates its compatibility with forms and accessing data, as repeatables is used to define lists of items, where each item has the same set of fields, but the user can add/remove multiple entries dynamically. In this example, staffCosts is a table-like structure where each row corresponds to a team member.

It’s separate from fields because fields are for single-instance fields (like "Project Title" or "Summary") that appear once per form, whereas repeatables are for arrays of entries.

Also with forms, it renders nicely:

  • id → used in react-hook-form as the array key (staffCosts), so that each item maps to staffCosts[0], staffCosts[1], etc.

  • title → displayed above the array block in the form UI.

  • itemSchema → defines the fields for each entry in the array. Every item has exactly the same shape: name, role, hours, rate.

This being used at a higher level will be the schema being loaded into a DynamicFormPage of some sort that renders it in the section.

Decision 5 — Data Table Integration with shadcn/ui

Context:

The application requires a responsive, accessible, and easily maintainable data table component to display dynamic datasets (e.g. project listings, costing breakdowns, or user submissions). The table should integrate seamlessly with the overall UI design system and support features like pagination, sorting, and filtering if needed.

Options Considered:

  • Option A — Build a fully custom table component

  • Option B — Use a prebuilt table library (e.g. React Table, TanStack Table)

  • Option C — Use the shadcn/ui DataTable pattern built on TanStack Table with Tailwind

Decision:

  • Adopt the DataTable component pattern from shadcn/ui for all tabular data displays.

  • Wrap the table component in a custom utility for easy loading of data from external sources or API responses.

Consequences:

  • Significant reduction in development time with prebuilt table logic (sorting, pagination, filters).

  • Consistent styling aligned with the rest of the design system.

  • Easy to integrate with backend or local data sources by passing data and column definitions as props.

  • Minor learning curve for TanStack Table API and shadcn’s wrapper pattern.

Decision 6 — Page Rendering Strategy

Context:

Application requires a workspace shell with persistent navigation (sidebar, header, right panel) while swapping content in the main area. Need predictable routing for developer-controlled pages containing forms, tables, and interactive components. Pages may contain internal tabs with independent views. Future may require dynamic dashboards or admin-configurable layouts.

Options Considered:

  • Option A — Direct component rendering with array-based routing for all pages

  • Option B — Block-driven content renderer with JSON schema for dynamic layouts

  • Option C — Hybrid approach using direct components for core pages, blocks for configurable sections

Decision:

  • Use direct component rendering with array-based routing for workspace pages

  • Implement persistent WorkspaceShell that maintains header, and state

  • Define pages in central route array mapping path to component

  • Handle tabbed content within pages using local state and conditional rendering

  • Preserve tab state using hidden divs rather than unmounting components

  • Keep schema-driven forms and reusable components compatible with direct rendering

Consequences:

  • Predictable, type-safe routing and page composition for developers

  • Lower complexity overhead compared to block-driven system for MVP

  • Workspace state preserved across navigation (no sidebar/header re-renders)

  • Tab state maintained within pages for better user experience

  • Clear migration path to hybrid approach when dynamic layouts become necessary

  • Reusable components (forms, tables) work seamlessly with direct rendering

Notes:

  • Data Table documentation: https://ui.shadcn.com/docs/components/data-table

  • Pages can be registered in a central array for routing:

// routes.ts or could be simple array in the WorkspaceShell
export const projectRoutes = [
  { path: "overview", component: ProjectOverviewPage },
  { path: "staff-costs", component: StaffCostsPage },
  { path: "budget", component: BudgetPage },
  // add more pages here
];

  • Workspace shell loads the active route component:

<WorkspaceShell>
  <CurrentPageComponent />
</WorkspaceShell>

  • Tabs inside pages are optional and handled locally:

<TabBar tabs={["Budget", "Staff"]} selectedTab={tab} onChange={setTab} />
<div hidden={tab !== "Budget"}><BudgetForm /></div>
<div hidden={tab !== "Staff"}><StaffList /></div>

  • This pattern allows reusing schema-driven forms, tables, and other UI components without introducing the complexity of a full block/JSON-driven system.

  • Keeps routing and workspace logic simple while allowing later migration to block-driven pages if needed.

Decision 7 — Page Content Layout and Responsive System

Context:

Pages within the WorkspaceShell need consistent layout structure, spacing, and responsive behaviour without requiring developers to manually implement containers, breakpoints, or spacing logic for each page. Need separation between global layout (workspace shell) and page-level content organization while ensuring visual consistency across all pages.

Options Considered:

  • Option A — Manual layout implementation per page with custom CSS and breakpoints

  • Option B — Utility-first approach using only Tailwind classes directly in each page

  • Option C — Standardised layout primitive components (PageContent, PageSection, PageRow) with built-in responsive behaviour

Decision:

  • Implement standardised layout primitive system with X amount of core components

  • PageContent serves as root container with vertical stacking, consistent padding, and max-width constraints

  • PageSection groups related content blocks with optional titles

  • PageRow handles horizontal layouts that automatically collapse to single column on mobile

  • Use Tailwind responsive utilities at container level rather than component level

  • Maintain separation between workspace shell (global), page layout (primitives), and component styling (internal)

Consequences:

  • Consistent visual appearance and spacing across all pages without repetitive layout code

  • Automatic responsive behaviour without manual breakpoint management per page

  • Clear separation of layout concerns makes pages easier to maintain and extend

  • Developers can focus on content and functionality rather than layout mechanics

  • Slight learning curve for layout primitive usage patterns

  • Risk of layout constraints if primitives don't cover edge cases

Notes:

An example outline of how it might work

PageContent is the root container for page content. It acts like an auto-layout frame in Figma, stacking blocks vertically, applying consistent padding, and constraining width.

PageRow is used to place two or more blocks side by side on larger screens, while automatically stacking them on mobile.

Decision 8 — Routing Architecture

Context:

Application requires clear separation between public routes (login, password reset) and authenticated workspace routes (dashboard, projects, settings). Authenticated routes should share common workspace layout while public routes use minimal standalone layouts. Need efficient routing that minimises re-rendering of persistent UI elements.

Options Considered:

  • Option A — All routes use same layout with conditional navigation rendering

  • Option B — Separate routing trees with different layout wrappers per route type

  • Option C — Two-level nested routing with workspace shell for authenticated routes only

Decision:

  • Builds on Decision 5.

  • Implement two-level routing model with persistent workspace shell for authenticated sections

  • Public routes render as standalone components without workspace layout

  • All authenticated routes render inside persistent WorkspaceShell component

  • Use nested routing where WorkspaceShell wraps authenticated routes and provides Outlet

Consequences:

  • Clean separation between public and authenticated user experiences

  • Consistent workspace navigation across all main application pages

  • Minimal re-rendering since sidebar, header, and panels persist during navigation (inside workshell)

  • Easy to extend by adding routes to configuration without layout concerns

  • Global state preservation across workspace navigation

  • Slightly more complex routing setup with nested route structure

Notes:

Public routes: /login render standalone. Authenticated routes: /dashboard, /projects, /projects/:id render inside WorkspaceShell. Navigation between workspace routes only swaps main content via React Router Outlet. Route configuration includes layout type metadata. Internal page tabs handled by individual page components, not routing system. Supports future addition of routes without changing fundamental architecture.
