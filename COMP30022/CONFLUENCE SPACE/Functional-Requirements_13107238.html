Authentication & User Management

Requirement

Solution

Priority (H, M, L)

Users must be able to log in and access their saved projects.

Implement secure user authentication with session management and link each user account to their saved projects in the database.

H

Support user roles (candidate)

Role-based access control in the system

H

Support user roles (admin, approver)

Role-based access control in the system

H

Log in via UoM Single Sign-On

Integrate with UoM SSO

(out-of-scope)

Core System & Data Entry

Requirement

Solution

Priority (H, M, L)

Create web-based version of the Research Costing and Pricing tool

Replicate (most of the) existing Excel logic and linked data sources

H

Enable editable entry of project details (title, duration, resources, staff, consumables)

Web forms for data entry, editable until approved

H

Input compulsory and optional information boxes

Prevent continuation or submission until all complete

H

Support attachment of documents (e.g., grant proposal PDFs), or some form of additional/other field

File upload functionality

L

Add an undo/back button to reverse edits

Store staged JSON versions of project data to allow reverting to previous states

L

Admin & System Configuration

Requirement

Solution

Priority (H, M, L)

Update base data (e.g., salary rates)

Admin-only interface to edit and update reference data tables (similar to Lookup Table in current workbook)

H

Configure internal rates and key system parameters

Admin interface for changing internal properties

M

Configure workflow steps to add / remove steps or change order (make extendable)

Admin interface to define approvers or sequence or change steps. Do not make it like a glorified Google Forms.

M

Provide RIC admins with a dashboard and system-wide visibility

Admin dashboard / training documentation

M

Change rates, entries, and other aspects of applicants form.

Admin interface to update salary/indirect rates and other entries

L

Costing & Calculations

Requirement

Solution

Priority (H, M, L)

Calculate total project cost automatically using internal rates from backend data

Implement calculation engine based on salary and indirect rates

H

Allow pricing adjustments (margin, discount)

Input fields or sliders to adjust price

H

Be able to show the Budget Currency and adjust depending on the currency selected, or enter a raw exchange rate.

Integrate a conversion date and conversion rate tool

M

Annual financial data must be provided/imported to allow calculation of the Cost Recovery Multiplier

Integrate data entry with the calculation engine for the Cost Recovery Multiplier

M

Perform compliance checks against relevant policies

Check cost calculations against defined rules with the Policy sheet mentioned

L

Refresh the currency exchange rate

Update periodically with the new exchange rate

L

Approvals & Workflow

Requirement

Solution

Priority (H, M, L)

Track approval workflow for each project

Approval status tracking in system OR stage attribute for a user/application (e.g. User → HoD → Faculty → notification back) and asked for stage visibility)

H

Maintain centralised access for approved budgets

Role-based access to approved documents, and keep tool internal.

H

Track status of projects through stages (draft, submitted, approved)

Saved to a logged in user and stored OR saved as Excel / external file to be loaded in each time. This keeps it OR status field in database with visual indicators

H

External/public approval view for approvers (e.g., HoD, Dean)

Generate a secure, read-only overview link for specific approvers to view and approve projects without logging in

M

Users need a notes/feedback section to communicate with RIC admin about issues

Implement a dedicated notes/feedback field within the project record, separate from approval comments optionally link to admin notifications

M / H

Allow approvers to send comments/questions to candidates

Comment interface tied to project record

L

Allow candidates to “respond” to comments or queries (either with checkbox or some form of communication)

Reply interface in same project record

M

Notify approvers of pending approvals

Email notifications or require shareable approval view of project via secure link.

M

Approval stages should be visible to users and admins

Add a pipeline/step-tracker UI showing all approval stages and current progress

M

Support visibility of approvals/comments across faculties where applicable

Project record shows all approvers’ comments/status

M

Track communication history for auditing

Log messages and responses within project record

L

Notify users of system updates or deadlines

Email alerts or dashboard notifications

L

Staff Costs & Non-Staff Costs

Requirement

Solution

Priority (H, M, L)

In staff costs, have the required fields, which have predetermined selections populated

Store plausible selections.

H

In staff costs, depending on fields enter, populate the rate the staff should be expecting

Integrate up to date employment rates depending on qualifications, integrate this with some Government site or UniMelb data

H

Have an “in-kind contributions” (which are no attributed to direct costs) section of the in staff costs

have different sections or different fields depending on the type of staff or type of employment

H

Create the oncosts summary view, with an adjustable, and overridable UoM oncosts percentage

Summary view created based off data entered form the earlier forms.

H

Create non-staff costs view with correct fields and attributes.

Make a similar interface for the sections with these additional attributes.

H

In staff costs, Having a rolling total Cost of the project displayed at all times, with two different multipliers (the one selected, and the universities recommended multiplied)

Constantly update this total cost of the project throughout the application

H

In staff costs, breakdown a timeline of payment and staff time

Auto fill the timeline based on prior filled information

M

Group costs by type in a different summarised view for simplicity (typically non-staff costs)

implement an appropriate relational DB that can group by certain attributes.

M

Budget & Forms

Requirement

Solution

Priority (H, M, L)

“Budget form” is a section to summarise all details and filled in appropriately for a admin to see, approve, or make recommendations

Have a separate interface of the summarised information, and all the data correctly formatted

H

In budget form (part c and d), have other mandatory sections which need to eb filled in based off the Faculty / school, typically signatures

Support multiple users adding required information/signatures to their designated sections

H

Display non-compliant budget if it would not even be looked at based on the regulations

Set up necessary hurdles for the budget to adhere to like the minimum cost multiplier or if there are any in-kind contributions.

H

TR1F form section for multi-faculty and disciplines specifications and with appropriate attributes.

TR1F form section should appear only when required for multi-faculty/discipline projects

M
