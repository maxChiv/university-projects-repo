Group Twenty Two Project : Product Backlog

Created by Max Chivers, last modified on Aug 21, 2025

Purpose

The Product Backlog contains all project deliverables, features, and requirements. This is the source of truth for all work items and is used to populate sprint backlogs which the Scrum Master will collate before a sprint planning session.

How to Use This Page

  • All new requirements or ideas should be added here first.

  • Each backlog item should have clear attributes filled in.

  • Items can later be selected for a sprint backlog during sprint planning.

  • Use labels to indicate priority, type, or sprint suitability.

  • This is a master Trello Board which will be filtered into sprints.

1. Functional Requirements Specification

Possible Tasks:

  • Review existing Excel tool and note all features

  • Draft list of functional requirements

  • Prioritise requirements (H/M/L)

  • Circulate for team feedback

2. Low-Fidelity Wireframes

Possible Tasks:

  • Create rough wireframes for main screens (login, project entry, cost summary)

  • Map navigation flow between pages

  • Share wireframes for team review

  • Adjust based on feedback

3. User Flow / Interaction Diagram

Possible Tasks:

  • Draft a flow diagram showing user interactions (candidate, admin, approver)

  • Identify decision points and approval steps

  • Label inputs, outputs, and system calculations

  • Review with team for completeness

4. Data Entry Form Design

Possible Tasks:

  • Sketch forms for project details, staff costs, non-staff costs

  • Include mandatory vs optional fields

  • Annotate validation rules (required fields, data types)

  • Align with functional requirements

5. Calculation Engine Design

Possible Tasks:

  • Document Excel formulas to replicate

  • Decide on backend structure to perform cost calculations

  • Identify data sources and mapping to web fields

  • Plan adjustable parameters (rates, multipliers, margins)

6. Report & Export Design

Possible Tasks:

  • Sketch layout for PDF/Excel reports

  • Decide which sections are mandatory and optional

  • Define fields that auto-populate vs editable by admin

  • Identify approval indicators for reporting

7. Role-Based Access & Workflow Design

Possible Tasks:

  • Define user roles (candidate, admin, approver)

  • Document which screens/functions each role can access

  • Sketch approval workflow stages

  • Annotate notifications & comments flow

8. System Architecture / Tech Stack Decision

Possible Tasks:

  • Decide on backend/frontend technologies

  • Document database schema or tables needed

  • Map external integrations (SSO, Excel, policies)

  • Review scalability, maintainability, and security considerations

9. UI/UX Guidelines

Possible Tasks:

  • Decide colour scheme, fonts, and layout standards

  • Annotate accessibility considerations

  • Define style for tables, forms, and buttons

  • Share draft with team for feedback

10. Compliance & Security Requirements (High-Level)

Possible Tasks:

  • List compliance rules (data privacy, internal policies)

  • Highlight design decisions that address compliance

  • Identify areas needing admin control

  • Review with team
