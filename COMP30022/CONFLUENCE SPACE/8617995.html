Group Twenty Two Project : 📘 Project Delivery Framework

Created by Max Chivers on Aug 19, 2025

Written with collaboration with ChatGPT.

This page outlines how we manage tasks, backlogs, and sprints in our project. It sets the expectations for what a “task” should look like and how we progress through our 4×2 sprint schedule.

🔄 Agile Approach

We are following an Agile-inspired method, tailored to our 8-week timeline.

  • 4 × 2-week sprints

  • Use of Kanban board to manage tasks

  • Documentation maintained in Confluence

  • Client engagement and feedback at the end of each sprint

🗂️ Backlog Structure

  • Product Backlog:

    • The master list of deliverables for the project (features, designs, documentation, deployments).

    • Updated continuously as requirements change.

  • Sprint Backlog:

    • A subset of backlog items chosen for the current sprint.

    • Owned by the whole team during sprint planning.

    • Each item must be small enough to complete within the sprint.

    • These should be prioritised.

✅ What is a Task?

A task is the smallest unit of work we track on the Kanban board. Tasks are created to complete backlog items.

A good task should have:

  • Clear description – what exactly needs to be done.

  • Owner – who is responsible.

  • Deliverable / Output – tangible outcome (e.g., document, diagram, code snippet).

  • Status – To Do, In Progress, Done.

  • Effort size – should be small enough to complete in 1–2 days.

Examples:

  • Backlog item: “Design system architecture.”

    • Tasks: “Draft architecture diagram in Lucidchart,” “Review design with team,” “Upload diagram to Confluence.”

  • Backlog item: “Implement login use case.”

    • Tasks: “Set up authentication routes,” “Implement user model,” “Test login flow.”

📅 Sprint Schedule

Sprint 1 (Week 5–6): Design + Research

  • Deliverables: UI wireframes, database schema, architecture diagram, tech stack decision.

Sprint 2 (Week 7–8): Core Coding

  • Deliverables: Implement base use cases, set up system skeleton.

Sprint 3 (Week 9–10): Extended Coding + Documentation + Testing

  • Deliverables: Additional use cases, updated Confluence docs, testing.

Sprint 4 (Week 11–12): Deployment & Finalisation

  • Deliverables: Deployed system, final documentation, client presentation.

👥 Task Workflow

  1. Define Tasks

    • Sprint planning → backlog items broken into tasks.

    • Tasks added to Kanban board.

  2. Assign Tasks

    • Team members volunteer or are assigned tasks in weekly meeting.

    • Each task must have a clear owner.

    • A good idea would be to work in pairs for consulting.

  3. Work on Tasks

    • Move through stages: To Do → In Progress → Done.

    • Update board regularly.

  4. Review & Deliver

    • Sprint review = show completed deliverables to client.

    • Retrospective = reflect on process, improve for next sprint.

🏁 Stages of a Sprint

  1. Sprint Planning (start of sprint)

    • Team + product owner/client decide what goes into the sprint backlog.

    • Break backlog items into tasks on the Kanban board.

    • Goal: set the sprint commitment.

  2. During Sprint

    • Daily Stand-ups (or check-ins)

      • Short, ~15 min.

      • Each person: what I did, what I’ll do, blockers.

      • Frequency: ideally daily, but for student teams every few days works fine.

      • Goal: sync the team, not solve everything.

  3. Sprint Review (end of sprint)

    • Audience: client + team.

    • The team demos what’s done — working features, designs, docs, etc.

    • Focus: the product. Did we deliver what we promised?

    • Goal: gather client feedback to adjust the backlog for next sprint.

  4. Sprint Retrospective (end of sprint, after review)

    • Audience: team only.

    • Reflects on the process, not the product.

    • Questions like:

      • What went well?

      • What didn’t go well?

      • What should we change for next sprint?

    • Goal: continuous improvement in teamwork.

🧩 Expectations

  • Every task must be visible on the Kanban board.

  • Each backlog item must have clear deliverables.

  • Confluence must be kept up to date with design decisions, meeting notes, and documentation.

  • Team members are responsible for updating progress and communicating blockers.
