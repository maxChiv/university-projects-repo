Group Twenty Two Project : Our Development Best Practices

Created by Max Chivers, last modified on Sept 10, 2025

This document outlines our team’s conventions and workflow for collaborating on this project. Following these practices will help us keep the codebase clean, consistent, and reliable.

Below each section there will be some additional links for similar reading.

Overview of Workflow:

We will follow a similar workflow suggesting in the lectures.

1. Coding Standards & Style Guide

  • Naming Conventions:

    • Use camelCase for variables and functions.

    • PascalCase for React components and Java classes.

    • Consistent and descriptive naming is expected.

  • Linting/Formatting:

    • Frontend: ESLint + Prettier.

    • Backend: Follow standard Java conventions, use Checkstyle or Spotless.

    • Code must pass lint checks before merging.

  • Google Style Guides

  • ESLint Getting Started

  • Prettier Documentation

  • Spring Boot Code Style Guidelines

2. Branching Strategy

  • Main branches:

    • main: stable production-ready code.

    • develop: integration branch for features/bugfixes.

  • Feature branches:

    • Name format: feature/<card-id>-<short-description>

      • Example: feature/19-user-authentication

  • Bugfix branches:

    • Name format: bugfix/<card-id>-<short-description>

      • Example: bugfix/20-login-null-pointer

  • Flow:

    • Feature/bugfix branches → Pull Request into develop.

    • develop → Pull Request into main at sprint end or major release.

*** Note :* Card ID can be found in the Trello URL link

3. Pull Request Guidelines

  • PR Template: Required.

  • Reviewers: At least 1 reviewer from the team (anyone).

  • Content:

    • Keep PRs meaningful and focused.

    • PR description is flexible — include anything useful (linked issue/Trello card, screenshots, notes).

  • Merging:

    • Only after all checks pass and review approval is given.

Code Review Guidelines

What to Look For:

  • Design: Is the code well-designed and appropriate for your system?

  • Functionality: Does the code behave as the author likely intended? Is the way the code behaves good for its users?

  • Complexity: Could the code be made simpler? Would another developer be able to easily understand and use this code when they come across it in the future?

  • Tests: Does the code have correct and well-designed automated tests?

  • Naming: Did the developer choose clear names for variables, classes, methods, etc.?

  • Comments: Are the comments clear and useful?

  • Style: Does the code follow our style guides?

  • Documentation: Did the developer also update relevant documentation?

What Reviewers Should Write in a PR Review:

  • Summary: High-level statement (e.g., “Code is clear and works as intended, just some minor style fixes”).

  • Inline Comments: Point to specific lines where issues/improvements can be made.

  • Categorise Feedback:

    • Required Changes (must be fixed before merge)

    • Suggestions (nice-to-have improvements)

  • Approval or Request Changes: Use GitHub’s built-in review options

4. Commit Message Standards

  • Use Conventional Commits format:

    • feat: for new features.

    • fix: for bug fixes.

    • chore: for non-functional changes (deps, config).

    • wip: for work-in-progress small steps

    • docs: for documentation.

    • refactor: for structural changes without behaviour changes.

  • Example:

    • feat: add user authentication service

    • fix: resolve login null pointer exception

5. Versioning & Release Notes

  • Versioning: Use Semantic Versioning (MAJOR.MINOR.PATCH).

  • Releases:

    • Releases created at the end of each sprint.

    • Tag releases in GitHub (v1.0.0, v1.1.0, etc.).

  • Release Notes:

    • Written manually by release manager (rotating role).

    • Summarise features, fixes, and breaking changes.

6. Branch Protection

  • main branch:

    • Protected: only merge from develop.

    • Requires passing CI checks.

    • No force pushes.

  • develop branch:

    • Protected: only merge from feature/bugfix branches.

    • Requires passing CI checks.

    • No force pushes.

Guide: Branch Protection

7. Issue Management & Workflow

  • Tool: Trello for task tracking.

  • Workflow:

    • Each Trello card represents a feature, bug, or task.

    • Branches/PRs should reference the card where possible.

    • Cards moved to “Done” once merged into develop.

    • Issues remain open until the fix/feature is verified in staging.

  • Templates:

    • Bug Report: steps to reproduce, expected vs actual.

    • Feature Request: description, acceptance criteria.

Github About issues

8. Additional Best Practices

  • Secrets: Never commit API keys or passwords — use environment variables.

  • Documentation: Update README and any relevant docs when behaviour changes.

  • Responsiveness: Aim to review PRs within 1–2 days.

  • Code Ownership: Informal — any team member can review any code.
